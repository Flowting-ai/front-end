### Phase 2 — Knowledge file size fix

**Goal:** One-line fix that unblocks correct information display.

#### 2.1 — KB/MB display

**Files:** `knowledge/page.tsx:63` and `knowledge/page.tsx:117`

**Fix the `docsToFiles` function:**
```js
// Replace:
const size = doc.size_bytes ? `${(doc.size_bytes / 1024 / 1024).toFixed(1)} MB` : '-'

// With:
function formatFileSize(bytes: number): string {
  if (bytes < 100 * 1024) return `${Math.round(bytes / 1024)} KB`
  return `${(bytes / 1024 / 1024).toFixed(1)} MB`
}
const size = doc.size_bytes ? formatFileSize(doc.size_bytes) : '-'
```

**Also fix `docsToFilesWithSizes`** — the inline `bytes / 1024 / 1024` on line 117:
```js
// Replace:
const withSize = bytes != null ? { ...f, size: `${(bytes / 1024 / 1024).toFixed(1)} MB` } : f

// With:
const withSize = bytes != null ? { ...f, size: formatFileSize(bytes) } : f
```

**Important constraint:** This does NOT create new versions, does NOT call the API. It only fires after sessionStorage write and returns the changed field names for the toast.

### Phase 4 — API deduplication

**Goal:** Eliminate redundant network requests.

#### 4.1 — Remove double `getVersion` call (context + instructions)

**Current:** `context.tsx` bootstrap effect calls `getVersion` to extract `model_id` for the guide panel. `instructions/page.tsx` `initialise()` also calls `getVersion`.

**Fix:** Remove the bootstrap effect from `context.tsx` (lines 193–231). Instead, have the Instructions tab push the guide model info into context via `updatePersonaInfo` after it loads the version — which it already does on line 769–780. The context bootstrap was added as a fallback for when the user opens a non-instructions tab directly (e.g. profile tab URL). 

**Better fix:** Keep the context bootstrap but use a shared in-flight promise pattern — if `versionId` is in the URL params on mount, do one fetch in context, store result, and share it to Instructions tab via a context value `versionBootstrap: PersonaVersionResponse | null`. Instructions tab reads from it instead of re-fetching.

**Simpler fix (no architecture change):** Cache the result in sessionStorage (`persona_version_cache_{repoId}_{versionId}`) with a 60s TTL. Both callers hit the cache. First call goes to network, second hits cache.

#### 4.2 — Remove redundant `getVersion` in Profile tab

**File:** `profile/page.tsx:176–183`

This block fetches `getVersion` just to set `currentPrompt` (used for guide). The Instructions tab already calls `updatePersonaInfo({ guidePrompt: instruction })`. The guide prompt is already in context.

**Fix:** Delete the `getVersion` effect in profile. Replace `currentPrompt` with `personaInfo.guidePrompt` from the shared context. No API call needed.

```js
// Remove:
useEffect(() => {
  if (!repoId || !versionId) return
  getVersion(repoId, versionId)
    .then(v => { setCurrentPrompt(v.prompt ?? '') })
    .catch(() => {})
}, [repoId, versionId])

// And instead, when you need the guide prompt, just read it from context:
const { personaInfo } = usePersonaConfigure()
// personaInfo.guidePrompt is already set by Instructions tab
```

#### 4.3 — Reduce `getPersonaRepo` calls in Profile tab

**Current:** Profile tab calls `getPersonaRepo(repoId)` even when a meaningful sessionStorage draft exists.

**Fix:** Keep the API call (it's needed for avatar refresh and for the first-edit case), but only call it when there is NO meaningful draft. When `hasMeaningfulDraft = true`, skip the API call entirely. Avatar can fall back to the context's `personaInfo.imageUrl` which was set by Instructions.

```js
if (hasMeaningfulDraft) {
  // Still refresh avatar from context if not in draft
  if (!avatarUrl) setAvatarUrl(personaInfo.imageUrl)
  isInitializedRef.current = true
  setIsInitialized(true)
  return
}
// Only fetch API if no meaningful local draft
getPersonaRepo(repoId).then(...)
```

### Phase 7 — Wizard starter call with tone (enhancement)

**Goal:** Pass the selected tone to the starter API so it influences the generated system instruction.

**Current:** `personaStarter({ name, description })` is called on the Name page (no tone). The tone is picked on the next page and stored but never passed to any API.

**Problem:** The generated system instruction doesn't reflect the user's tone choice.

**Fix options:**

**Option A (Recommended):** Move the starter call to the Tone page continue handler. Change the call to include tone:
```js
await personaStarter({ name, description: purpose, tone: selectedTone })
```
This means the Tone page continue has a loading state while the API runs. The starter response (system_instruction + tags) is stored in `persona_wizard_starter` as before.

**Downside:** The tone page now has a loading spinner on continue, adding ~1-2s latency before entering configure.

**Option B:** Keep the call on the Name page (pre-fetch for snappiness), but call it again with the tone after the user selects. This doubles the API calls.

**Option C:** Keep the Name page call (no tone) but store the selected tone in `persona_wizard_draft`. On the Instructions page, use the stored tone to enhance/re-color the pre-generated instruction. This is already partly done (tone is stored) but nothing reads it for instruction generation.

**Recommended: Option A** — correctness over snappiness. The user already waits on the tone page to read tone options; waiting an extra ~1-2s on continue is acceptable.

**Changes:**
1. Remove `personaStarter` call from `name/page.tsx`. Remove loading state from name page.
2. Add `personaStarter` call to `tone/page.tsx` `handleContinue` with `{ name: draft.name, description: draft.purpose, tone: selectedTone }`.
3. The tone page already has a Continue button that navigates — just add loading spinner during the API call.

### Phase 8 — Leave guard: save draft on leave

**Current:** Clicking "Leave" in the leave confirm modal just navigates away. No draft is explicitly saved.

**Spec says:** Leave → save current state as a draft version → navigate to `/personas`.

**Context / layout changes:**
- The leave guard modal currently has "Stay" and "Leave" buttons.
- "Leave" should: (1) call the registered auto-save function, (2) navigate.
- The auto-save function for Instructions writes to sessionStorage — this is already a draft. The persona already exists as a draft in the backend from the initial `createPersonaRepo` call.
- So "save as draft" is already implicitly handled — sessionStorage draft exists and the repo exists. No new API call is strictly needed.
- What IS needed: ensure the `/personas` library correctly shows the draft card.

**Verdict:** The leave logic is mostly correct. The draft card appears because `persona_needs_publish_{repoId}` is set in localStorage at repo creation. Fix needed: ensure the leave modal "Leave" button clears the leave confirm state AND navigates properly (check that the navigation actually fires after leave, not silently fails).

This is a **low priority verification** — check the layout's leave modal "Leave" button handler to confirm the navigation works.
